## Vista funcional estructurada
| Tema | Explicacion en NFR-001 |
|---|---|
| Capacidades funcionales del producto | Condiciona capacidades del producto para cumplir: Tiempo de respuesta de APIs core. |
| Funciones principales | Aplica a endpoints core del flujo de catalogo y compra en MVP. |
| Comportamientos esperados | p95 <= 800 ms en lecturas de catalogo y p95 <= 1500 ms en mutaciones de carrito/pedido.; validado mediante: Pruebas de carga por endpoint + metricas runtime semanales. |
| Casos de uso de alto nivel | Aplica a flujos operativos de alto nivel cubiertos por el alcance operativo del requisito. |

## Cobertura de temas NFR con impacto semantico
| Tema requerido | Cobertura explicita en NFR-001 |
|---|---|
| Restricciones de negocio con impacto operativo | Protege la operacion comercial en flujos core al fijar presupuesto de respuesta para lectura de catalogo y mutaciones de carrito/pedido. |
| Condiciones de rendimiento o disponibilidad que alteran diseno | Aplica de forma directa: define umbrales de rendimiento p95 que condicionan arquitectura de consultas, cache y capacidad. |
| Aislamiento, regionalizacion, seguridad, compliance, trazabilidad | No es foco primario de aislamiento/regionalizacion/seguridad/compliance; aporta trazabilidad tecnica via medicion continua y evidencia de cumplimiento. |
| NFR que cambia la manera en que el sistema debe ser concebido | Cambia el diseno al exigir presupuesto por endpoint y observacion continua de latencia para decisiones de capacidad. |

## Detalle de la restriccion (tabla)
| Tema | Detalle |
|---|---|
| Enunciado | Tiempo de respuesta de APIs core. |
| Metrica y umbral | p95 <= 800 ms en lecturas de catalogo y p95 <= 1500 ms en mutaciones de carrito/pedido. |
| Verificacion y validacion | Pruebas de carga por endpoint + metricas runtime semanales. |
| Alcance operativo | Aplica a endpoints core del flujo de catalogo y compra en MVP. |
| No alcance | No aplica a endpoints no-core ni a procesos batch de larga duracion. |
| Supuestos y restricciones | La carga de prueba debe representar el comportamiento real de uso del MVP. |
| Preguntas abiertas | No hay preguntas abiertas bloqueantes para `MVP`; el baseline de carga queda definido en calidad/arquitectura y su ajuste fino por ventana horaria es `No bloqueante`. |
| Impacto esperado | Optimizacion de consultas, cache selectiva y presupuestos por endpoint. |
