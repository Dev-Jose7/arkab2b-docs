## Detalle del requisito (tabla)
| Tema | Detalle |
|---|---|
| Descripcion | El sistema autentica usuarios y autoriza operaciones por organizacion y rol de negocio. |
| Racional | Sin aislamiento por organizacion y perfil de acceso no hay operacion B2B segura. |
| Alcance | - Autenticacion y autorizacion por organizacion y rol de negocio.<br>- Rechazo funcional de operaciones no autorizadas con trazabilidad.<br>- Baseline RBAC minimo del `MVP` para habilitar pruebas y seguridad operativa:<br>  - owner de organizacion (`ORG_OWNER`)<br>  - usuario de organizacion (`ORG_USER`)<br>  - administrador de plataforma (`ARKA_ADMIN`)<br>- El catalogo de roles puede extenderse sin romper el baseline, conservando semantica de `RoleCode` y controles de acceso. |
| No alcance | - Federacion de identidad empresarial avanzada (SSO complejo).<br>- Administracion IAM externa completa fuera de MVP. |
| Actor(es) | - Usuario B2B y sistema de control de acceso. |
| Disparador | - Inicio de sesion o intento de operacion protegida. |
| Precondiciones | - Cuenta registrada con membresia organizacional y rol asignado.<br>- Politicas de permiso funcionales configuradas para el MVP. |
| Flujo principal | 1. El sistema autentica credenciales de la cuenta.<br>2. Resuelve contexto organizacional y perfil de acceso efectivo.<br>3. Evalua permiso para la operacion solicitada.<br>4. Autoriza o rechaza segun reglas, dejando traza de sesion/evento. |
| Postcondiciones | - Operaciones validas se ejecutan bajo aislamiento por organizacion.<br>- Intentos no autorizados quedan registrados para auditoria. |
| Supuestos y restricciones | - La matriz granular puede evolucionar por pais y segmento sin romper el baseline minimo del MVP.<br>- El detalle tecnico de enforcement se define en otro pilar.<br>- En este baseline IAM no se modela como multi-tenant; el aislamiento se expresa por organizacion y perfil de acceso. |
| Preguntas abiertas | - Ninguna bloqueante para el baseline `MVP`; la granularidad fina por pais/segmento queda para evolucion posterior. |
| Criterios de aceptacion | - Cuenta autenticada opera solo en su contexto organizacional autorizado.<br>- Operacion no autorizada se rechaza con error de negocio.<br>- Sesion y permisos auditables.<br>- MFA para cuentas administrativas Arka (`ARKA_ADMIN`) queda fuera del baseline<br>  actual y se difiere a hardening posterior no bloqueante. |
| Dependencias | - NFR-005<br>- NFR-006 |
| Riesgos | - Configuracion incompleta de permisos por rol/pais. |
| Notas de implementacion | La matriz granular de permisos por pais/segmento no forma parte del baseline operativo actual del MVP. |
