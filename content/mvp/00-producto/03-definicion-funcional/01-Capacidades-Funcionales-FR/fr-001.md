## Detalle del requisito (tabla)
| Tema | Detalle |
|---|---|
| Descripcion | El sistema permite registrar y mantener productos y variantes (SKU) con atributos comerciales, usando taxonomia referencial activa de marca/categoria dentro de Catalog. |
| Racional | Sin catalogo confiable no hay compra digital eficiente. |
| Alcance | - Alta y edicion de productos y variantes (SKU) con atributos comerciales.<br>- Validacion funcional de obligatoriedad, unicidad de SKU por variante y existencia de marca/categoria activas como taxonomia referencial del catalogo. |
| No alcance | - CRUD independiente de marcas/categorias por API en `MVP`; la taxonomia se mantiene por carga administrativa controlada del propio catalogo.<br>- Definicion del modelo de dominio interno de catalogo.<br>- Motor avanzado de busqueda, ranking o recomendaciones. |
| Actor(es) | - Administrador u operador de catalogo. |
| Disparador | - Solicitud de crear o actualizar un producto/variante desde backoffice. |
| Precondiciones | - Usuario autenticado y autorizado para gestion de catalogo.<br>- Marca y categoria activas dentro de la taxonomia referencial del catalogo. |
| Flujo principal | 1. El actor ingresa datos comerciales del producto/variante.<br>2. El sistema valida campos requeridos y unicidad de SKU.<br>3. El sistema registra o actualiza la informacion.<br>4. El sistema confirma el resultado al actor. |
| Postcondiciones | - Producto/variante queda disponible para operaciones posteriores del MVP.<br>- Queda registro trazable de la accion realizada. |
| Supuestos y restricciones | - El MVP prioriza consistencia funcional sobre capacidad masiva de catalogo.<br>- La definicion fina de atributos queda sujeta a validacion de negocio. |
| Preguntas abiertas | - No hay preguntas abiertas bloqueantes para `MVP`; la lista fina de atributos por categoria queda como `Evolucion posterior` sin cambiar el baseline funcional de catalogo. |
| Criterios de aceptacion | - Se registra nombre, descripcion, marca/categoria referenciales validas y atributos requeridos.<br>- SKU unico por variante.<br>- Confirmacion de alta/edicion. |
| Dependencias | - FR-009 |
| Riesgos | - Datos maestros incompletos. |
| Notas de implementacion | Pendiente validacion de atributos finales con negocio.<br>El pricing se gobierna dentro del mismo bounded context, pero fuera del alcance principal de CRUD descrito en este requisito. |
