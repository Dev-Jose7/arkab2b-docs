## Detalle del requisito (tabla)
| Tema | Detalle |
|---|---|
| Racional | Permite operación MVP sin pasarela online completa. |
| Alcance | - Registro manual de pagos y actualización del estado de pago del pedido.<br>- Validación de monto y referencia no duplicada. |
| No alcance | - Integración de pasarela de pago online y conciliación automática bancaria.<br>- Gestion de contracargos o antifraude avanzado. |
| Actor(es) | - Operador financiero/comercial autorizado. |
| Disparador | - Recepción de pago y solicitud de registro sobre un pedido. |
| Precondiciones | - Pedido existente y elegible para registrar pagos.<br>- Referencia de pago disponible y monto mayor a cero. |
| Flujo principal | 1. El actor registra referencia y monto del pago.<br>2. El sistema válida formato y duplicidad de referencia.<br>3. Registra el pago si la validación es exitosa.<br>4. Recalcula estado de pago (`PENDING`, `PARTIALLY_PAID`, `PAID`). |
| Postcondiciones | - Pago manual trazado en el sistema.<br>- Estado de pago del pedido actualizado de forma consistente. |
| Supuestos y restricciones | - Alcance limitado al flujo manual definido para MVP.<br>- Reglas técnicas de idempotencia fina se completan en otro pilar. |
| Preguntas abiertas | - No hay preguntas abiertas bloqueantes para `MVP`; la corrección avanzada de pagos queda como `Evolucion posterior` sobre el baseline de registro manual idempotente. |
| Dependencias | - FR-004<br>- NFR-006 |
| Riesgos | - Errores humanos en registro de pago. |
| Descripción | El operador registra pagos manuales y el estado de pago del pedido se actualiza. |
| Tipos de pago manual compatibles (sets) | \| Set                            \| Tipos incluidos                                                         \| Uso esperado                                                               \|<br>\|--------------------------------\|-------------------------------------------------------------------------\|----------------------------------------------------------------------------\|<br>\| Set A - Operativo ampliado     \| `TRANSFERENCIA_BANCARIA`, `CONSIGNACION_BANCARIA`, `EFECTIVO`, `CHEQUE` \| Operación manual con mayor cobertura de casos de recaudo.                  \|<br>\| Set B - Operativo estricto B2B \| `TRANSFERENCIA_BANCARIA`, `CONSIGNACION_BANCARIA`                       \| Operación manual con menor variabilidad y menor riesgo operativo en `MVP`. \| |
| Criterios de aceptación | - Pago requiere referencia y monto válido (>0).<br>- No se permite duplicado por referencia.<br>- El estado de pago del pedido se recalcula usando `PENDING`, `PARTIALLY_PAID` o `PAID`.<br>- Los tipos de pago manual permitidos deben declararse usando uno de los sets compatibles (`Set A` o `Set B`), sin introducir pago online en `MVP`. |
| Notas de implementación | Alcance limitado a pago manual en MVP. |
